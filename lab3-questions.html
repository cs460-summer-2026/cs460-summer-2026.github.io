Lab 3 — TA Prep Sheet

CS460 Summer 2026 · Run the VERSUS PA1 skeleton · Likely student questions and quick answers

How this lab runs: students self-serve from the lab page. Your opening line: "Have a look at Lab 3 and follow the steps. Each step has a green 'Check it worked' box, don't move on until your output matches. If something goes wrong, the Troubleshooting section probably has it. Otherwise just ask. Once you can register and create a bracket in the browser, you're free to go."
Three things to announce up front to the whole room (preempts most questions):
  • Windows: use PowerShell, not the old Command Prompt. Some commands won't work in cmd.
  • Mac: if you hit "port 5000 in use," it's AirPlay Receiver. Either turn it off (System Settings → General → AirDrop & Handoff) or change the port in app.py to 5001.
  • Apostrophe in your username will break registration. The skeleton SQL is teaching code, it doesn't handle quotes. Use a username without an apostrophe.

Setup & environment questions

I unzipped samplecode.zip but cd samplecode and there's no app.py.

Windows extraction often creates a nested folder, samplecode\samplecode\app.py. Have them dir and cd one level deeper, or drag the inner folder up. Troubleshooting covers this.

I'm in the right folder but python app.py says "no such file."

They're not actually in the folder. Have them run pwd (Mac) or Get-Location (Windows) to confirm where they are, then ls/dir to confirm app.py is visible.

My venv prompt doesn't show (cs460env) after activating.

Three possibilities: (1) Windows: PowerShell execution policy blocked the script, see the execution policy fix. (2) They typed the activate command wrong, the Windows path uses backslashes cs460env\Scripts\Activate.ps1. (3) They're in a venv they made elsewhere, fine, just re-create or re-activate the right one.

PowerShell error: "running scripts is disabled on this system."

Standard PowerShell block. Run Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser, press Y. One-time, then activate works. This is the biggest Windows trap, expect it a lot.

"externally-managed-environment" when running pip.

Venv isn't active. Prompt must show (cs460env). Activate again, then retry pip.

Do I need to use the venv from Lab 2 or make a new one?

Reuse Lab 2's if they can find it. Otherwise make a new one with python3 -m venv ~/Desktop/cs460env. Either works, the lab uses Desktop path to stay consistent with Lab 2.

Schema & database questions

PowerShell complains: "The '<' operator is reserved for future use."

PowerShell doesn't support < for input redirection. Use Get-Content schema.sql | mysql -u root -p instead, or load schema.sql in Workbench (which works the same on any OS).

Loading schema.sql gives "Access denied."

Wrong root password. Retype it. If they're sure it's right, they may be confusing their MySQL root password with their Mac/Windows login password, those are different.

SHOW TABLES is empty after loading the schema.

They didn't run USE versus; first. The schema creates the database but you have to switch into it to see its tables. If SHOW DATABASES; doesn't list versus at all, the schema actually failed, scroll back through their terminal for the real error.

I already had a database called versus from somewhere else and now it's gone.

Yes, schema.sql starts with DROP DATABASE IF EXISTS versus;. That's intentional for fresh starts. Tell them to recreate whatever they had, or rename it next time.

Running the app questions

"ModuleNotFoundError: No module named 'flask'" (or 'mysql').

Venv isn't active, or they installed packages outside the venv. Activate the venv (prompt shows (cs460env)), reinstall pip install -r requirements.txt, then python app.py. All three from the same activated shell.

"mysql.connector.errors.ProgrammingError: 1045 Access denied for user 'root'."

Their DB_PASSWORD in app.py doesn't match their real MySQL root password. Open app.py, fix the password between the single quotes, save, re-run. Blank password is DB_PASSWORD = '' (two quotes, nothing between, no spaces).

"Unknown database 'versus'."

Schema didn't run. Go back to Step 3 and load it. Confirm with SHOW DATABASES; before retrying.

"Address already in use" on port 5000.

Either they have a stale Python process running, find that terminal and Ctrl+C, or change the last line of app.py to app.run(port=5001, debug=True) and use localhost:5001 in the browser. Mac specifically: this is almost always AirPlay Receiver, see the next question.

Mac: port 5000 is "AirPlay Receiver."

macOS Monterey and newer reserved port 5000 for AirPlay. Two options: (1) System Settings → General → AirDrop & Handoff → turn AirPlay Receiver off. (2) Change Flask to a different port in app.py. Option 1 is cleaner because the lab and the rest of the project assume port 5000.

"jinja2.exceptions.TemplateNotFound: hello.html"

Running app.py from the wrong directory. The templates/ folder has to sit next to app.py in their current shell. Have them pwd/Get-Location, then cd back into the skeleton folder.

The terminal returns to my prompt after running python app.py.

The server crashed. Scroll up in the terminal for the actual error. It's almost always one of: wrong DB_PASSWORD (1045), wrong database name (Unknown database), or port in use. The traceback names the cause.

Browser & usage questions

Browser says "This site can't be reached."

Three checks: (1) python app.py still running in their terminal? (2) Did they type http:// not https://? (3) Did they change the port to 5001 but go to 5000? Most "can't be reached" is one of those three.

Register fails with "You have an error in your SQL syntax."

They used an apostrophe in username or password (e.g. O'Brien). The skeleton uses naive string interpolation in SQL, single quotes break it. Tell them this is a known limitation of the teaching code and to use a name without an apostrophe. We fix this with parameterized queries later in the course.

Create-bracket form won't submit, no error.

They changed the entrant count dropdown after partially filling in the form, so some seed fields are empty but still required. The browser silently blocks submission. Tell them to scroll the form and fill every Seed line.

My bracket has 7 matchups but only 4 show entrant names, the rest say (TBD).

That's correct. Round 1 has all 4 matchups filled in (8 entrants paired). Rounds 2 and 3 are empty shells waiting to be filled in by the close_round stored procedure they'll build for PA1. Not a bug, expected output.

I changed app.py but my edit isn't showing in the browser.

Debug mode usually auto-reloads, but big edits sometimes don't. Ctrl+C the server, run python app.py again, reload the browser. (Also: did they save the file? Common miss.)

Higher-level questions you might get

Why does the SQL use string formatting like "...'{0}'".format(x)? Isn't that bad?

Yes, it's SQL injection vulnerable, and it's intentional for teaching simplicity. The README acknowledges this. We'll cover parameterized queries (cursor.execute(sql, params)) in a later lab. For PA1 they can keep this style, or switch to parameterized if they're comfortable.

Why is the DB connection a single global variable?

Same reason, teaching simplicity. A real Flask app uses a connection pool or per-request connections. For their PA1 the single global is fine, it just means if MySQL restarts mid-session the app needs a restart too.

Do I need to commit the venv folder?

No, never. Venv folders are huge and machine-specific. Add it to .gitignore if they're using git. Same for any IDE folders like .vscode or .idea.

Am I done with Lab 3?

Point them at the "You're done when…" checklist. Four items: SHOW TABLES lists four tables, python app.py runs without errors, register works, 8-entrant bracket creates and displays 7 matchups. Once all four are true, lab is over.

If a student is genuinely stuck for more than 5 minutes on something not in the page's Troubleshooting, the likely culprits in order: (1) venv not actually active, prompt missing the prefix, (2) wrong directory, (3) wrong password in app.py, (4) Workbench connected but they're running the script connected to a different MySQL. Pair-up stuck students with the ones who finished.
