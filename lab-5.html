0. The one big idea

This whole lab is just one idea repeated:

Cost of an operation  =  how many PAGES it reads or writes from disk.

  read 1 page  = 1 I/O
  write 1 page = 1 I/O

Reading from disk is the slow part of a database, much slower than anything the CPU does. So when we ask "how expensive is this query?" we just count page reads and writes and ignore everything else. Every number in this lab is a count of pages. That's the entire game.

Two words we'll use the whole time:

  • N or M = the size of a table, measured in pages (not rows).
  • B = the size of our buffer pool, i.e. how many pages of memory we get to use at once.

1. External Sorting

The situation

We need to sort a table, but it is way bigger than memory. We can't load it all in and call sort(). So we sort it in passes, reading and writing the file to disk one pass at a time.

  • Pass 0: read the file B pages at a time, sort each chunk in memory, write it back out. Each sorted chunk is called a run.
  • Later passes: merge runs together into bigger runs. With B buffers we can merge B−1 runs at once (1 buffer is saved for the output). We keep merging until there is just one run left, which is the fully sorted file.

Two facts are all you need:

number of passes = 1 + ⌈ log(B-1) of ⌈N/B⌉ ⌉

total cost = 2N × (number of passes)

Why 2N per pass? Because every pass reads all N pages once and writes all N pages once. Read + write = 2N. Simple.

Don't memorize the log formula yet. Just watch the example below count passes by hand, then the formula will feel obvious.

Worked example

Sort a file of N = 108 pages using B = 5 buffer pages.

Pass 0: read 5 pages at a time, sort each, write back.
        108 / 5 = 21.6  →  round up to 22 runs of 5 pages each.
        (we now have 22 sorted runs on disk)

Pass 1: merge B-1 = 4 runs at a time.
        22 / 4 = 5.5  →  6 bigger runs.

Pass 2: merge 4 at a time again.
        6 / 4 = 1.5  →  2 runs.

Pass 3: merge those 2 runs into 1.
        DONE. 1 fully sorted run.

Count the passes: 0, 1, 2, 3 → that's 4 passes.

Check with the formula:
  1 + ⌈ log4 of ⌈108/5⌉ ⌉
= 1 + ⌈ log4 of 22 ⌉
= 1 + 3            (because 4³ = 64 ≥ 22, but 4² = 16 < 22)
= 4 passes   ✓

Total I/O cost = 2N × passes
              = 2 × 108 × 4
              = 864 I/Os.

That's the whole topic. Count how many passes it takes to get down to 1 run, then multiply 2N × passes.

2. Join Costs

A join matches rows from two tables that share a value (for example, match each reservation to the sailor who made it). There are several ways to do it, and they cost wildly different amounts of I/O. We'll count the cost of three of them on the same two tables.

Our two tables (memorize these numbers)

R = Reserves :  M = 1000 pages,  100 rows per page
S = Sailors  :  N =  500 pages,   80 rows per page

We are joining R and S. In all three methods below, one table is the outer (the one in the outer loop) and the other is the inner (re-scanned again and again).

Method 1 — Simple Nested Loops (the slow one)

For every single row of the outer table, scan the entire inner table. Notice it says every row, not every page — that's what makes it terrible.

Cost = (rows in outer) × (pages in inner) + (pages in outer)

With R outer:
  rows in R = 1000 pages × 100 rows = 100,000 rows
  Cost = 100,000 × 500 + 1000
       = 50,001,000 I/Os    ← about 50 million. Awful.

Fifty million page reads. This is the version we never want.

Method 2 — Page Nested Loops (a bit smarter)

Same idea, but loop over pages instead of rows. For every page of the outer, scan all pages of the inner.

Cost = (pages in outer) × (pages in inner) + (pages in outer)

With R outer:
  Cost = 1000 × 500 + 1000 = 501,000 I/Os

With S outer (smaller table outside):
  Cost = 500 × 1000 + 500 = 500,500 I/Os

From 50 million down to ~500 thousand just by looping over pages, not rows. And putting the smaller table on the outside is slightly better. (Good rule of thumb: smaller relation outer.)

Method 3 — Block Nested Loops (the practical one)

The trick: use most of our memory to hold a big block of the outer table at once. If we can hold 100 outer pages in memory, then we only scan the inner table once per block of 100 pages, instead of once per single page. Far fewer scans of the inner.

Cost = (pages in outer) + (number of outer blocks) × (pages in inner)

number of outer blocks = ceil( pages in outer / block size )

Say our block holds 100 pages of the outer table.

With R outer (1000 pages, block = 100):
  number of blocks = 1000 / 100 = 10
  Cost = 1000 + 10 × 500 = 6000 I/Os

With S outer (500 pages, block = 100):
  number of blocks = 500 / 100 = 5
  Cost = 500 + 5 × 1000 = 5500 I/Os

From 500,000 down to ~6000. Same data, same join — just a smarter use of memory. This is why the buffer pool size B matters so much: a bigger block means fewer inner scans.

Common mistakes

"I counted rows when I should have counted pages"

Cost is measured in pages (I/Os), not rows. The only place rows show up is Simple Nested Loops, where the outer loop runs once per row — and that is exactly why it costs millions.

"I forgot to round up"

You always round up when counting runs or blocks, because a leftover half-page still needs a whole pass / a whole block. 22/4 = 5.5 means 6 runs, never 5.

"I put the wrong table on the outside"

For nested-loop joins, the smaller table goes on the outside. Fewer outer pages (or blocks) means fewer scans of the inner table.

"My pass count is off by one"

Pass 0 counts as a pass. Count 0, 1, 2, 3… until you are down to a single run. If the file fits in B pages from the start, that's just Pass 0 = 1 pass.

"I used B instead of B−1 when merging"

Pass 0 uses all B buffers to sort. But every merge pass keeps one buffer for output, so it merges only B−1 runs at a time.
