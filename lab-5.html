0. The Cost Model

Every cost in this lab is measured the same way:

Cost of an operation  =  number of PAGES it reads or writes from disk.

  read 1 page  = 1 I/O
  write 1 page = 1 I/O

Reading and writing disk pages is far slower than any work the CPU does, so it dominates the running time of a query. For that reason we estimate cost by counting page reads and writes and ignore CPU time. Every figure in this lab is a count of pages.

Two symbols are used throughout:

  • N or M: the size of a table, measured in pages (not rows).
  • B: the size of the buffer pool, that is, how many pages of memory are available at one time.

When a problem gives you a table as a number of tuples instead of pages, your first step is always to convert tuples into pages. Section 3 shows how.

1. External Sorting

The problem

We need to sort a table that is larger than memory, so it cannot be loaded in full and sorted at once. Instead we sort it in passes, reading and writing the whole file to disk on each pass.

  • Pass 0: read the file B pages at a time, sort each chunk in memory, and write it back. Each sorted chunk is called a run.
  • Later passes: merge runs into larger runs. With B buffers we can merge B−1 runs at once, because one buffer is reserved for the output. We keep merging until a single run remains, which is the sorted file.

Two formulas are all you need:

number of passes = 1 + ⌈ log(B-1) of ⌈N/B⌉ ⌉

total cost = 2N × (number of passes)

The cost is 2N per pass because every pass reads all N pages once and writes all N pages once, so read plus write equals 2N.

You do not need to memorize the log formula. The worked example below counts the passes by hand, and the formula simply confirms that count.

Worked example

Sort a file of N = 108 pages using B = 5 buffer pages.

Pass 0: read 5 pages at a time, sort each, write back.
        108 / 5 = 21.6  →  round up to 22 runs of 5 pages each.
        (we now have 22 sorted runs on disk)

Pass 1: merge B-1 = 4 runs at a time.
        22 / 4 = 5.5  →  6 larger runs.

Pass 2: merge 4 at a time again.
        6 / 4 = 1.5  →  2 runs.

Pass 3: merge those 2 runs into 1.
        DONE. 1 sorted run.

Count the passes: 0, 1, 2, 3, which is 4 passes.

Check with the formula:
  1 + ⌈ log4 of ⌈108/5⌉ ⌉
= 1 + ⌈ log4 of 22 ⌉
= 1 + 3            (4³ = 64 ≥ 22, but 4² = 16 < 22)
= 4 passes   ✓

Total I/O cost = 2N × passes
              = 2 × 108 × 4
              = 864 I/Os.

That is the whole method: count how many passes it takes to reach one run, then multiply 2N × passes.

Exercise: external sorting

Same method, smaller numbers. Sort a file of N = 80 pages with B = 5 buffer pages.

How many runs does Pass 0 make?

Show answer

Pass 0 sorts B = 5 pages at a time.

80 / 5 = 16 → 16 runs of 5 pages each.

How many total passes? (keep merging 4 at a time until 1 run)

Show answer 
Pass 0: 16 runs
Pass 1: 16 / 4 = 4 runs
Pass 2: 4 / 4  = 1 run  → DONE

Passes 0, 1, 2, which is 3 passes.

Formula check: 1 + ⌈log4 of 16⌉ = 1 + 2 = 3. ✓

What is the total I/O cost?

Show answer

2N × passes = 2 × 80 × 3 = 480 I/Os.

2. Join Costs

A join matches rows from two tables that share a value, for example matching each reservation to the sailor who made it. There are several algorithms for it, and they differ greatly in I/O cost. We count the cost of three of them on the same two tables.

The two tables

R = Reserves :  M = 1000 pages,  100 rows per page
S = Sailors  :  N =  500 pages,   80 rows per page

We are joining R and S. In each method below, one table is the outer (the outer loop) and the other is the inner (scanned again for each step of the outer loop).

Method 1: Simple Nested Loops

For every row of the outer table, scan the entire inner table. Note that it is every row, not every page, which is why the cost is so high.

Cost = (rows in outer) × (pages in inner) + (pages in outer)

With R outer:
  rows in R = 1000 pages × 100 rows = 100,000 rows
  Cost = 100,000 × 500 + 1000
       = 50,001,000 I/Os    (about 50 million)

About fifty million page reads. This version is never used in practice.

Method 2: Page Nested Loops

The same idea, but the loops run over pages instead of rows. For every page of the outer, scan all pages of the inner.

Cost = (pages in outer) × (pages in inner) + (pages in outer)

With R outer:
  Cost = 1000 × 500 + 1000 = 501,000 I/Os

With S outer (smaller table outside):
  Cost = 500 × 1000 + 500 = 500,500 I/Os

Looping over pages instead of rows brings the cost from about 50 million down to about 500 thousand. Putting the smaller table on the outside is slightly cheaper. A useful rule of thumb is: smaller relation outer.

Method 3: Block Nested Loops

Use most of the buffer to hold a large block of the outer table at once. If the block holds 100 outer pages, the inner table is scanned once per block of 100 pages rather than once per single page, which reduces the number of inner scans.

Cost = (pages in outer) + (number of outer blocks) × (pages in inner)

number of outer blocks = ceil( pages in outer / block size )

Suppose the block holds 100 pages of the outer table.

With R outer (1000 pages, block = 100):
  number of blocks = 1000 / 100 = 10
  Cost = 1000 + 10 × 500 = 6000 I/Os

With S outer (500 pages, block = 100):
  number of blocks = 500 / 100 = 5
  Cost = 500 + 5 × 1000 = 5500 I/Os

The cost drops from about 500,000 to about 6000 on the same join. This is why the buffer size B matters: a larger block means fewer scans of the inner table.

Exercise: join costs

Same two tables. Reminder:

R = Reserves :  M = 1000 pages,  100 rows per page
S = Sailors  :  N =  500 pages,   80 rows per page

Page Nested Loops with S as the outer table. What is the cost?

Show answer

Cost = (outer pages) × (inner pages) + (outer pages).

500 × 1000 + 500 = 500,500 I/Os.

Block Nested Loops with R as the outer and a block that holds 200 pages. What is the cost?

Show answer 
number of blocks = 1000 / 200 = 5
Cost = 1000 + 5 × 500 = 1000 + 2500 = 3500 I/Os

A larger block (200 instead of 100) gives a lower cost than the 6000 from earlier. More memory means fewer inner scans, which means lower cost.

Which is cheaper for Block Nested Loops: R outer or S outer (both with a 100-page block)? Why?

Show answer

R outer = 6000, S outer = 5500, so S outer is cheaper.

The smaller table (S) forms fewer blocks, so the larger table is scanned fewer times. Smaller relation outer wins again.

3. Exam-style Problem

Final-exam join questions do not hand you the page counts. They give you table statistics and a page size, and you convert them to pages yourself before costing the plan. This exercise walks through that full process on a small schema.

The setup

Schema:
  Employee(eid: integer, name: char(20), dept: char(10))
  Project (pid: integer, eid: integer, hours: integer)

Sizes:
  integer = 4 bytes,  char(n) = n bytes
  page size = 100 bytes
  buffer pool B = 12 pages

Statistics:
  Employee : 2000 tuples
  Project  : 6000 tuples
  eid is the primary key of Employee and a foreign key in Project

We want the cost of joining Employee and Project on eid using a Block Nested Loop Join.

Step 1. Tuple size, then tuples per page

Add up the bytes in one tuple, then see how many whole tuples fit in a 100-byte page. A tuple cannot be split across pages, so round the tuples-per-page down.

How many tuples fit on one page for each table?

Show answer 
Employee tuple = 4 (eid) + 20 (name) + 10 (dept) = 34 bytes
  tuples per page = floor(100 / 34) = floor(2.94) = 2

Project tuple  = 4 (pid) + 4 (eid) + 4 (hours)   = 12 bytes
  tuples per page = floor(100 / 12) = floor(8.33) = 8

Round down because a partial tuple cannot be stored on a page.

Step 2. Pages per table

Divide total tuples by tuples per page, and round up (a leftover handful of tuples still needs one more page).

How many pages does each table occupy?

Show answer 
Employee : 2000 / 2 = 1000 pages   →  call this M = 1000
Project  : 6000 / 8 = 750  pages   →  call this N =  750

Now the problem looks like the join problems from Section 2, just with our own page counts.

Step 3. Block Nested Loop cost

Use the smaller table as the outer, with a block that holds B−2 = 10 pages (one buffer for the inner, one for the output).

Put Project (the smaller table, 750 pages) as the outer with a block of 10 pages. What is the cost?

Show answer 
number of blocks = ceil(750 / 10) = 75
Cost = (outer pages) + (blocks) × (inner pages)
     = 750 + 75 × 1000
     = 750 + 75,000
     = 75,750 I/Os

If instead Employee (1000 pages) were the outer: ceil(1000 / 10) = 100 blocks, cost = 1000 + 100 × 750 = 76,000 I/Os. Project outer is slightly cheaper, matching the smaller-relation-outer rule.

Step 4. Propose a cheaper plan

Exam questions often ask you to improve the plan. The standard improvement is to replace the nested-loop join with a Hash Join (or Sort-Merge Join), whose cost is 3 × (M + N) when the smaller table fits the memory condition.

What is the Hash Join cost, and how much does it save?

Show answer 
Hash Join cost = 3 × (M + N)
              = 3 × (1000 + 750)
              = 3 × 1750
              = 5250 I/Os

That is about 75,750 / 5250 ≈ 14 times cheaper than the Block Nested Loop plan. The partitioning phase reads and writes both tables (2(M+N)), and the matching phase reads both once more (M+N), giving 3(M+N) in total.

This is the same shape as Problem 2 on a CS460 final: convert statistics to pages, cost a Block Nested Loop plan, then propose a Hash Join that is roughly ten times cheaper or more.

Common mistakes

Counting rows when the cost is in pages

Cost is measured in pages (I/Os), not rows. The only place rows appear is Simple Nested Loops, where the outer loop runs once per row, which is exactly why that method costs millions.

Rounding the wrong way

Tuples per page rounds down, because a partial tuple cannot fit on a page. Pages per table, runs, and blocks all round up, because a leftover remainder still needs a whole page, run, or block. For example, 22/4 = 5.5 means 6 runs, never 5.

Putting the wrong table on the outside

For nested-loop joins, the smaller table goes on the outside. Fewer outer pages or blocks means fewer scans of the inner table.

Off-by-one on the pass count

Pass 0 counts as a pass. Count 0, 1, 2, 3, and so on, until a single run remains. If the file already fits in B pages, that is Pass 0 only, so 1 pass.

Using B instead of B−1 when merging

Pass 0 uses all B buffers to sort. Every merge pass keeps one buffer for output, so it merges only B−1 runs at a time. For Block Nested Loops, the outer block holds B−2 pages, since one buffer is for the inner and one for the output.
