Read this first

Today is practice. Theory is in the lecture slides, today you actually run the algorithms by hand. Two short walkthroughs (one BCNF decomposition, one B+ tree of 16 keys), each with steps you try yourself before clicking to reveal the answer.

Multiple correct answers exist in both parts. If your decomposition or your tree differs from the reveal but follows the rules, you're probably right. The Common mistakes section at the bottom explains when "different" actually means "wrong".

Quick reference

Functional dependencies

  • X → Y means: any two rows that agree on X also agree on Y.
  • Attribute closure X⁺: start with X, then keep adding right-hand sides of any FD whose left-hand side is already in X⁺.
  • X is a superkey if X⁺ contains every attribute of R.
  • X is a candidate key if it's a superkey and no proper subset of X is a superkey.

BCNF

R is in BCNF if for every non-trivial FD X → Y, X is a superkey.

If some FD violates this, decompose: replace R with R - Y and XY.

B+ tree

Every node holds between d and 2d keys. In this lab d = 2, so every node holds between 2 and 4 keys. Root may hold fewer.

  • Insert: find the leaf, put the key in. If the leaf now has 5 keys, split it.
  • Leaf split: half the keys stay, half move to a new leaf. The smallest key of the new leaf is copied up to the parent.
  • Index node split: the middle key is pushed up to the parent (not copied).
  • If the root splits, the tree grows one level taller.

1. Normal Forms

Given a relation R = (A, B, C, D) and the following FDs:

AB → C
B  → D
C  → A

Decompose R into BCNF.

Step 1. Find the candidate key

Try each likely subset of attributes. Compute its closure. If the closure contains all of ABCD, it's a superkey. Minimal superkeys are candidate keys.

Try first

What is (AB)⁺? Is AB a candidate key?

Show answer

Start with {A, B}. Apply AB → C, add C. Apply B → D, add D. Now {A, B, C, D}. So (AB)⁺ = ABCD, AB is a superkey.

Is it minimal? Check the singletons. A⁺ = A (no FD has just A on the left). B⁺ = BD. Neither is a superkey, so AB is a candidate key.

Also try (BC)⁺: start with B and C, get D from B → D, get A from C → A. So (BC)⁺ = ABCD and BC is also a candidate key.

Candidate keys: AB and BC.

Step 2. Find a BCNF violation

For each FD, ask: is the left-hand side a superkey?

Try first

Which of AB → C, B → D, C → A violate BCNF?

Show answer
  • AB → C: AB is a candidate key (so a superkey). OK.
  • B → D: B⁺ = BD, not a superkey. Violates BCNF.
  • C → A: C⁺ = AC, not a superkey. Violates BCNF.

Two violations. Pick one and decompose.

Step 3. Decompose

Take the violation B → D. Replace R with R - {D} and BD.

Try first

Write the two new relations and the FDs that hold on each.

Show answer

R1 = (A, B, C) with FDs from the original projected onto ABC: AB → C and C → A.

R2 = (B, D) with FD B → D.

R2 is in BCNF (B is the key of R2 and the only FD has B on the left). R1 is not done yet, C → A still violates BCNF.

Step 4. Handle the remaining violation

In R1, C → A still violates BCNF. Decompose R1 the same way.

Try first

Decompose R1 using C → A. What are the final relations?

Show answer

R1 = (A, B, C) splits into R1a = (B, C) and R1b = (A, C).

Final BCNF decomposition: (A, C), (B, C), (B, D).

Sanity check: each relation has its key on the left of every FD that holds inside it. Done.

Step 5. Is it lossless?

A two-way split R → X, Y is lossless if X ∩ Y is a key of X or of Y.

Try first

Check each of the two splits we made.

Show answer

First split: R → (A, B, C) and (B, D). Intersection is {B}. B is the key of (B, D). Lossless.

Second split: (A, B, C) → (A, C) and (B, C). Intersection is {C}. C is the key of (A, C) (since C → A). Lossless.

Both splits lossless, so the whole decomposition is lossless.

Multiple correct decompositions exist. If you started with C → A first, you'd end up with a different (still valid) set of relations.

2. B+ Trees

Insert these 16 keys into an empty B+ tree in the given order. Each node holds up to 4 keys.

49, 24, 54, 31, 16, 76, 85, 68, 15, 40, 13, 35, 67, 18, 17, 29

The tree starts empty. The first 4 inserts just fill a single leaf:

After 49, 24, 54, 31:

  [ 24 | 31 | 49 | 54 ]

Step 1. Insert 16, first split

The leaf is full. Adding 16 makes 5 keys, so split.

Try first

Split the leaf, copy the right half's smallest key up to a new root, and draw the tree.

Show answer 
After 16:

           [ 31 ]
           /    \
  [ 16 | 24 ]   [ 31 | 49 | 54 ]

The smallest key of the right leaf (31) is copied up. It still appears in the right leaf, that's correct for leaves.

Step 2. Insert 76, 85, 68

76 fits in the right leaf. 85 overflows it, so split. 68 then goes in the appropriate leaf.

Try first

Draw the tree after each of 76, 85, 68.

Show answer 
After 76:
           [ 31 ]
           /    \
  [ 16 | 24 ]   [ 31 | 49 | 54 | 76 ]

After 85: right leaf has 5 keys, split.
            [ 31 | 54 ]
           /    |     \
  [16|24]  [31|49]   [54|76|85]

After 68: 68 goes in the middle leaf (between 49 and 76 range).
            [ 31 | 54 ]
           /    |     \
  [16|24]  [31|49|68]   [54|76|85]

When a leaf splits, the smallest key of the right half (here 54) is copied up.

Step 3. Insert 15, 40

15 goes in the leftmost leaf, no overflow. 40 fills the middle leaf.

Try first

Draw the tree after 15 and 40.

Show answer 
After 15, 40:

              [ 31 | 54 ]
             /    |     \
  [15|16|24]  [31|40|49|68]   [54|76|85]

No splits, but the middle leaf is now full.

Step 4. Insert 13, 35

13 fills the leftmost leaf. 35 makes the middle leaf overflow.

Try first

Insert 13, then 35, and split the middle leaf.

Show answer 
After 13:
              [ 31 | 54 ]
             /    |     \
  [13|15|16|24]  [31|40|49|68]   [54|76|85]

After 35: middle leaf becomes [31|35|40|49|68], split.
Right half is [40|49|68], smallest key 40 is copied up.

              [ 31 | 40 | 54 ]
             /    |    |    \
  [13|15|16|24] [31|35] [40|49|68] [54|76|85]

The root now holds 3 keys, still room for one more.

Step 5. Insert 67, 18

67 fits in the third leaf (becomes full). 18 overflows the leftmost leaf and triggers an index split.

Try first

Insert 67. Then insert 18 and walk through the splits.

Show answer 
After 67:
              [ 31 | 40 | 54 ]
             /    |    |    \
  [13|15|16|24] [31|35] [40|49|67|68] [54|76|85]

After 18: leftmost leaf [13|15|16|18|24] has 5 keys, split.
Right half is [18|24], smallest key 18 copied up.
Root becomes [18 | 31 | 40 | 54], 4 keys, still OK.

              [ 18 | 31 | 40 | 54 ]
             /    |    |    |    \
  [13|15|16]  [18|24]  [31|35]  [40|49|67|68]  [54|76|85]

The root is now full. The next index-level overflow will split the root and grow the tree.

Step 6. Insert 17, 29

17 goes in [13|15|16] (now full). 29 goes in [18|24].

Try first

Insert 17 and 29. Does anything split?

Show answer 
After 17:
              [ 18 | 31 | 40 | 54 ]
             /    |    |    |    \
  [13|15|16|17] [18|24] [31|35] [40|49|67|68] [54|76|85]

After 29:  29 belongs in [18|24], becomes [18|24|29], no split.

Final tree:
              [ 18 | 31 | 40 | 54 ]
             /    |    |    |    \
  [13|15|16|17] [18|24|29] [31|35] [40|49|67|68] [54|76|85]

The tree finished at height 2 (root plus one leaf level) with all 16 keys placed. Different insertion orders give different trees.

If your tree differs from this one but every non-root node has 2 to 4 keys and the keys are in the right order, you almost certainly got it right too.

Common mistakes

"I got a different candidate key, am I wrong?"

Probably not. A relation can have several candidate keys. Compute the closure of yours, if it gives all attributes and no proper subset of it does, it's a valid candidate key. In the warm-up example both AB and BC are candidate keys.

"My BCNF decomposition has different relations than the reveal"

Also probably fine. Picking a different violating FD first gives different (still correct) final relations. Check that each of your final relations has its key on the left of every FD that holds inside it, that's the BCNF check.

"My decomposition is not dependency preserving"

It might not be. BCNF decompositions are always lossless but not always dependency preserving. If you need dependency preservation, fall back to 3NF (which keeps all FDs but allows some redundancy).

"I confused copy-up with push-up"

Leaf splits copy the dividing key up (it still appears in the leaf). Index node splits push the dividing key up (it leaves the original node). The first is for finding rows, the second is for routing.

"My B+ tree is taller than the reveal"

Probably fine if every non-root node has 2 to 4 keys. The exact height depends on whether borderline splits redistribute or actually split. As long as keys in left subtrees are smaller than the routing key and right subtrees are larger or equal, your tree is correct.

"I split a leaf into 1 and 4 instead of 2 and 3"

Standard practice is to split evenly. After splitting a 5-key leaf, the left half should have 2 keys and the right 3 (or vice versa). 1 and 4 violates the minimum (d = 2).
